# DGCE Controlled Write Contract

This document defines the DGCE controlled-write boundary between incremental planning and the scaffold writer.

The default contract is intentionally conservative. DGCE v1 creates new files only. It does not modify or overwrite existing files.

## Purpose

The controlled-write layer decides whether a planned file is allowed to reach `write_file_plan(...)`.

This layer is policy and orchestration metadata. The writer remains the final filesystem backstop against overwriting an existing file.

## Run Mode Contract v1

Workspace runs also persist a minimal descriptive `run_mode` field for traceability.

Read-only inspection helpers:

- `python -m dgce inspect <project_root>` reads persisted workspace artifacts for operator inspection.
- `python -m dgce explain <project_root> --section <section_id>` reads the persisted section output artifact only and formats a deterministic explanation from stored facts.
- Neither helper writes artifacts, recomputes authority, or performs generated reasoning.

- `create_only`
  - Corresponds to `allow_safe_modify=False`.
- `safe_modify`
  - Corresponds to `allow_safe_modify=True`.

`run_mode` is descriptive in v1. The actual enforcement behavior still comes from the existing controlled-write logic, ownership checks, identical-content guard, and the current internal flag plumbing.

Workspace runs also persist one normalized `run_outcome_class` for traceability.

Allowed values:
- `validation_failure`
- `execution_error`
- `partial_skipped_modify`
- `partial_skipped_ownership`
- `partial_skipped_ignore`
- `partial_skipped_exists_fallback`
- `partial_skipped_identical`
- `success_create_only`
- `success_safe_modify`

`run_outcome_class` is descriptive only in v1. It is derived deterministically from finalized run facts using a simple priority order:
- validation failure first
- execution error second
- then the first matching skip class in priority order
- otherwise success classified by `run_mode`

## Write Rules

For a planned file path, DGCE v1 applies the following rules:

- `create`
  - The file is allowed to reach the writer.
- `modify`
  - The file must be skipped.
- `ignore`
  - The file must be skipped.
- Not in change plan:
  - If the path is missing on disk, it is allowed to reach the writer.
  - If the path already exists on disk, it is skipped conservatively.

## Safe Modify v2

Safe Modify v2 is opt-in only. It is enabled explicitly through `run_section_with_workspace(..., allow_safe_modify=True)`.

When Safe Modify v2 is enabled:

- `create`
  - The file is written as in v1.
- `modify`
  - The file may be overwritten directly only if the path is already DGCE-owned.
- `ignore`
  - The file must still be skipped.
- Not in change plan:
  - If the path is missing on disk, it is written.
  - If the path already exists on disk, it is skipped conservatively.

Safe Modify v2 performs direct replacement only. It does not perform merges, diffs, patch application, or semantic reconciliation.

Safe Modify v2.5 adds a pre-write identical-content guard for explicit `modify` paths only:

- If the existing file bytes are exactly identical to the new generated bytes, the file is skipped.
- This skip is recorded as `decision: skipped` with `reason: identical`.
- If the bytes differ, the direct replace behavior from v2 proceeds unchanged.

Ownership enforcement for Safe Modify uses the persisted pre-run `.dce/ownership_index.json` only:

- If a `modify` path is not present in the pre-run ownership index, it is skipped conservatively.
- This skip is recorded as `decision: skipped` with `reason: ownership`.
- No ownership is inferred from filesystem contents.

## Current v1 Meanings

- `create`
  - The path appeared in the expected target set and does not currently exist in the scanned workspace.
- `modify`
  - The path appeared in the expected target set and already exists in the scanned workspace.
- `ignore`
  - The path exists in the scanned workspace but was not part of the expected target set.
- Not in change plan
  - The path is present in the current run's derived file plan but was not present in the persisted expected target set used to build the current change plan.

## Collision Handling

- DGCE skips known `modify` paths before the writer unless Safe Modify v2 is explicitly enabled.
- DGCE also skips `modify` paths during Safe Modify when they are not present in the pre-run ownership index.
- DGCE skips known `ignore` paths before the writer.
- DGCE also skips paths not in the change plan when those paths already exist on disk.
- `write_file_plan(...)` remains the final overwrite backstop and still raises `FileExistsError` if an existing non-authorized path reaches it unexpectedly.

## Ownership and Safety

- DGCE v1 does not overwrite existing files.
- DGCE v1 does not modify existing files in place.
- Safe Modify v2 may replace files only for explicit `modify` paths.
- Safe Modify ownership enforcement is conservative and read-only in v1.
- DGCE v1 does not perform diffs, merges, or patch application against existing project files.
- The writer only creates files that survived controlled-write filtering and are still absent on disk, unless an explicit Safe Modify overwrite path is authorized.

## Stability Expectations

- These rules are the intended DGCE write contract for v1 and Safe Modify v2.
- Any future relaxation beyond direct modify replacement requires an explicit new contract version.

## Write Decision Rules

`should_write_planned_file(...)` is a lightweight pre-check only.

Final per-path write decisions are determined in the controlled-write decision path and recorded in `write_transparency`, which is the authoritative run record.

Default rules:

- create → written
- modify → skipped
- ignore → skipped
- not in change plan:
  - missing on disk → written
  - existing on disk → skipped (conservative)

With `allow_modify_write=True`:

- create → written
- modify → written only when the path is DGCE-owned and not identical
- ignore → skipped
- not in change plan:
  - missing on disk → written
  - existing on disk → skipped (conservative)

## Write Transparency v1

DGCE v1 also records observational per-path write transparency for workspace runs.

This transparency is informational only. It does not affect planning, execution, advisory generation, or file writing behavior.

`write_transparency` is also the authoritative persisted record of the final per-path write decisions for a run.

Diff Visibility v1:

- Diff visibility is observational only.
- It is recorded only for `modify` paths that are actually written during Safe Modify.
- It includes byte counts and a lightweight `changed_lines_estimate`.
- `changed_lines_estimate` is approximate and deterministic; it is not a unified diff, patch, or semantic diff.

Recorded decision values:
- `written`
- `skipped`

Recorded reason values:
- `create`
  - The path was allowed to reach the writer and was written.
- `modify`
  - The path was classified as `modify`.
  - With default behavior it is skipped.
  - With Safe Modify v2 enabled it is written by direct replacement.
- `ignore`
  - The path was skipped because the change plan classified it as `ignore`.
- `exists_fallback`
  - The path was not present in the change plan, but already existed on disk and was skipped conservatively.
- `identical`
  - The path was classified as `modify`, Safe Modify was enabled, and the existing file bytes already matched the generated bytes exactly.
- `ownership`
  - The path was classified as `modify`, Safe Modify was enabled, and the path was not present in the pre-run ownership index.

Optional `diff_visibility` fields for modify-written entries:
- `before_bytes`
- `after_bytes`
- `changed_lines_estimate`

Bytes behavior:
- `bytes_written` is recorded only for files that were actually written.
- `bytes_written_total` is the deterministic sum of written file sizes only.
- Identical skips do not contribute to `bytes_written_total`.

Diff summary behavior:
- `modify_written_count`, `diff_visible_count`, `before_bytes_total`, `after_bytes_total`, and `changed_lines_estimate_total` include only modify-written entries with diff visibility.

## Ownership Tracking v1

DGCE also persists an informational `.dce/ownership_index.json` for files actually written in the current run.

- Ownership is derived from `write_transparency` only.
- Ownership output remains observational, but Safe Modify enforcement reads the persisted pre-run ownership index conservatively.
- Skipped files do not receive ownership entries.
